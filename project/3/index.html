專題 03 · Edge AI · Hardware Co-Design

基於輕量化 1D-CNN 之
硬體友善跌倒偵測系統

以單顆攝影機擷取人體骨架，透過一個極輕量的時間卷積網路判斷「跌倒 / 正常」， 並把同一個模型部署到 PC、Jetson Nano 與 FPGA（PYNQ-Z2）三種平台， 驗證它在邊緣裝置上的可行性、即時性與準確度。

  • 95.65%固定測試集準確率
  • 0.42 ms單次模型推論
  • 92.39%FPGA 上板準確率
  • ~60 FPSFPGA 推論吞吐
  • 1D-CNN
  • MediaPipe Pose
  • PyTorch / ONNX
  • INT8 量化
  • Vitis HLS
  • PYNQ-Z2
01 — 專題背景

為什麼要做這個題目？

跌倒是高齡者居家意外傷害的主要原因之一，能否在事件發生的當下即時偵測並發出警報， 往往決定了後續處置的黃金時間。常見的穿戴式裝置必須隨身配戴、容易被遺忘； 而純影像的深度模型雖然準確，運算量卻過大，難以塞進低功耗、低成本的邊緣裝置。

因此本專題選擇了一條折衷且務實的路線：先用 MediaPipe 把影像轉成人體骨架座標， 再用一個非常輕量的 1D-CNN 對「骨架的時間變化」做分類。 這樣模型不需要看原始畫面、運算量極小，同時保有足夠的辨識能力， 也讓「最後能不能被硬體加速」變成一個可以認真討論的問題。

核心命題

設計一個「硬體友善」的跌倒偵測模型 —— 在保持高準確率的前提下， 讓它小到能放進 FPGA，並用模型與硬體協同設計（Co-Design）的角度， 驗證它在 PC、Jetson、FPGA 三個平台上的表現差異。

02 — 系統設計

整體流程：從影像到警報

整個系統是一條清楚的資料流：影像串流 → 骨架擷取 → 時間序列張量 → 1D-CNN 分類 → 跌倒判斷。 骨架擷取與前處理放在 CPU，分類模型則是可被加速的核心，最後在三個平台上分別驗證。

資料與輸入規格

資料來源為 URFD 公開跌倒資料集。每段影片經 MediaPipe 抽出關鍵點後， 固定整理成 (36, 50) 的張量：

  • 36 維：18 個關鍵點的 (x, y) 座標
  • 50 幀：以滑動視窗取的時間長度
  • 座標已正規化到 0~1，偵測不到的點補 0
  • 測試集 92 筆，採 20% 切分、固定 seed=42
03 — 模型設計與概念

刻意「為硬體著想」的 1D-CNN

模型本身刻意保持簡單：兩個卷積區塊加上一個分類頭。但每一個設計選擇背後， 都同時考慮了「準確率」與「之後能不能上硬體」這兩件事。

硬體友善的關鍵取捨

  • 用 ReLU6 取代 ReLU：輸出有上界，量化時數值範圍更穩定。
  • 移除 Dropout：推論期無作用，且簡化硬體資料路徑。
  • Conv 不含 bias：減少參數與一組加法器，方便 HLS 實作。
  • 固定輸入尺寸：(36, 50) 讓硬體記憶體配置可預先決定。

為什麼選 1D-CNN？

跌倒的關鍵資訊在於「骨架隨時間的變化」。1D 時間卷積剛好擅長抓這種序列特徵， 且相較於 LSTM / Transformer，它的計算高度規律（大量規律的 MAC 運算）、 沒有遞迴依賴，非常適合在 FPGA 上做流水線化與平行化。

04 — 概念與取捨

最重要的決定：放棄 DPU，改做訂製加速器

專題中段遇到一個關鍵岔路。原本打算用 Xilinx 的 DPU + Vitis AI 在 FPGA 上跑模型， 但實測發現 PYNQ-Z2（Zynq-7020）資源不足以承載完整 DPU，這條路在這塊板子上行不通。 與其硬湊，我選擇重新定位 FPGA 的角色：不用通用加速器，而是親手設計一個訂製的 1D-CNN 加速器 IP。

原方案：DPU + Vitis AI

  • 黑盒式通用加速器，無法展現協同設計
  • IP 體積大，Z2 資源不足 — 實測不可行
  • 偏向「使用工具」而非「設計硬體」

採用：訂製 HLS 加速器

  • 模型結構為硬體而優化，真正的 Co-Design
  • 只加速最關鍵的卷積，資源需求可控
  • 用 AXI-DMA + AXI-Stream 串接，INT8 定點運算
05 — 遇到的困難

實作過程踩過的坑

把一個模型從 PC 一路推到 FPGA，真正花時間的往往不是模型本身，而是跨平台的各種細節問題。

PYNQ-Z2 跑不動 DPU

板子資源不足以承載完整 DPU IP。最後改為訂製單層加速器 + CPU 協同，反而讓專題更有設計深度。

Jetson 裝不了 MediaPipe

Jetson 環境無法安裝 mediapipe。解法：PC 先抽好骨架輸出成 .npy，Jetson 端直接讀骨架做推論。

Bash 腳本的 CRLF 問題

Windows 換行讓 Jetson 出現 /bin/bash^M 錯誤。用 dos2unix 轉換換行符即可解決。

輸出影片歪斜、過快

畫面被拉伸、播放速度異常。強制偶數尺寸、固定輸出 FPS 並統一解析度後修正。

PC / Jetson 比較資料不足

比較腳本找不到對應設定的資料。將 PC 端 warmup 設為 0，讓兩平台量測條件一致。

瓶頸不在模型而在前處理

剖析後發現骨架擷取佔了端到端 96% 的時間，使「加速模型」的方向必須重新思考（見下節）。

06 — 實驗結果

數據怎麼說

準確率與量化

指標數值
Accuracy95.65%
Precision93.33%
Recall93.33%
F1-score93.33%

INT8 量化後準確率維持 95.65%，與 FP32 相比 Delta = +0.0000 — 證明這個模型對量化非常友善，幾乎無損。

跳幀消融：準確率 vs. 速度

透過「跳幀」可以用準確率換取速度。Stride=1 時準確率 93%、約 21 FPS； 拉到 Stride=4 仍有 89% 準確率，FPS 卻翻倍到 46。這給了實際部署一個可調的甜蜜點。

  • Stride 1：93% / 20.97 FPS
  • Stride 4：89% / 46.13 FPS
  • Stride 8：84% / 76.18 FPS
  • Stride 16：75% / 84.25 FPS

系統瓶頸分析

把端到端流程拆成五個階段後，結論很明確： 骨架擷取（MediaPipe）佔掉約 96.71% 的時間， 而 1D-CNN 推論只佔 1.22%（約 0.42 ms）。

這個發現很重要 —— 真正的瓶頸是前處理與 I/O，而非模型本身。 也因此「進一步加速模型」對端到端 FPS 的邊際效益其實有限， 這正是硬體設計中最值得記錄的一課。

跨平台效能比較

平台模型推論延遲推論 FPS特性 / 優勢
PC (CPU)~0.42 ms~191（端到端）高吞吐，但高功耗、延遲不確定
Jetson Nano 2GB~0.42 ms~18（端到端）易開發，受 OS 排程影響
FPGA (PYNQ-Z2)0.41 ms（IP）/ 16.62 ms（含 DMA）~60.25確定性延遲、低功耗、即時性

值得注意的是，FPGA 上 16.62 ms 的延遲中，絕大部分（約 16.11 ms）來自 DMA 資料搬移， IP 真正的計算只佔約 0.41 ms。「推論 FPS 比較低」不代表部署價值較低 —— FPGA 的賣點是確定性延遲與低功耗，這跟通用平台是不同維度的取捨。

失效樣本分析

檢視模型判斷錯誤的案例，有助於理解它的極限。多數錯誤集中在「動作與跌倒高度相似」 或「骨架被嚴重遮擋 / 抽取不全」的邊界情況。

07 — 結論

總結與收穫

這個專題完整走過了「模型訓練 → 量化 → 跨平台部署 → 訂製 FPGA 加速器」的全流程， 並用實驗數據回答了最初的命題：一個刻意為硬體設計的輕量 1D-CNN， 確實能在邊緣裝置上維持高準確率與即時性。

  • 輕量 1D-CNN 在固定測試集達 95.65% 準確率，INT8 量化幾乎無損。
  • 同一模型成功部署到 PC、Jetson、FPGA 三平台並完成量測。
  • FPGA 上板準確率 92.39%，推論吞吐約 60 FPS，延遲穩定可預測。
  • 剖析證實瓶頸在骨架擷取而非模型，為後續優化指出正確方向。
  • 從 DPU 轉向訂製加速器，體現真正的模型–硬體協同設計。

最大的收穫

學到的不只是「怎麼把模型變小」，而是怎麼在「準確率、速度、功耗、硬體資源」之間做取捨， 以及如何用數據誠實地面對平台限制 —— 當 DPU 行不通時，重新定義問題反而打開了更有價值的方向。
